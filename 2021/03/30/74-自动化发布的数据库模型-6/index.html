¶ 权限管理数据表

权限管理涉及两个主要的数据表：action 及 permission。一个自动化软件发布，是不能缺少权限管理的。

我们设计的这个自动化软件发布平台，权限分为三大类：

  1. CREATE 权限：指定哪些用户可以拥有新建发布单及编译软件的权限。我们认为发布单新建及编译有同一个权限，谁新建的，当然有权限对它进行打包编译。这个权限主要针对研发人员设置的。
  2. XCHANGE 权限：指定哪些用户可以对应用进行环境流转权限。比如，从编译好的状态流转进 DEV 环境，TEST 环境，或是 PRD 环境。让测试人员知道现在研发的进度，并进行系统测试的配合。这个权限主要是针对测试人员设置的。
  3. DEPLOY 权限：指定哪个用户可以对哪个环境进行发布。比如，研发可以发布到 DEV 环境，而 TEST 环境和 PRD 环境谁可以发布，要视公司流程和规模而定。这样，每个人只对自己的环境负责。

在这三个权限之前，还有两个额外的系统权限需要注意：第一是系统的 Admin 超级权限，属于系统 Admin 组的人，具有应用的所有权限，且可以登录 Django 管理后台。这个 Admin 是 Django 本身的用户组，不用特别数据表支持。第二是每个服务应用的管理员，这个管理员可以进行各自服务内的三类权限分配。通过这样的授权，系统管理员就不会被粘住，应用管理员可以自助操作，增加了自动化的能力。这个角色，我们已经在之前的 App 数据表中的 manage_user 字段设置好了。

第8~12行，可以看到 Action 表和前面的 Env 表，deploy_status 表一样，都是为了方便扩展而设计的表，字段用 CommonInfo 类里面的字段，再加上一个 aid 的权限序号就可以满足。Action 表只是三行数据（CREATE，XCHANGE，DEPLOY）。

第28行，main_user，此字段为拥有上面所指定权限的用户。这是一个多对多的字段，表示这个字段里可以放置很多用户。由于这里没有使用 through 关键字，所以是隐式创建表。
