¶ 发布单状态数据表

发布单状态数据表，位于 deploy/models.py 文件中，这个表的作用，是将发布单的生命周期的各个状态独立出来。这个字段当然也可以合并进发布单数据表中，但是那样就不够灵活了。还有一个原因，一旦合并进发布单数据表，那么这个字段要么是中文，要么是英文，会让我们在写代码时有所制约。因为我们想实现的最佳实践是：发布单的状态为英文，但是在显示时为中文，且不同的应用可以定义为不同的名称。

基于以上原因，我们决定将发布单状态数据独立出来。它的状态数据分为 6 个：新建(CREATE)，编译(BUILD)，待发布(READY)，发布中(ING)，发布异常(ERROR)，发布完成(FINISH).

¶ 发布单数据表

发布单数据表也是位于 deploy/models.py 文件中，主要用于发布各种软件，每一次的发布，都会形成这条记录，并在新建，构建，流转，发布时，更新其中的字段。

¶ models.py 的文件内容

代码解释:
第8~11行，IS_INC_TOT_CHOICES，用于定义全量还是增量发布，这种元组的写法及配置 models.CharField(…, choices=IS_INC_CHOICES, …) 定义方法，可以让我们限定这个字段可能的值，并在后台以下拉框的方式呈现。
第13~18行，DEPLOY_TYPE_CHOICES，用于定义本次发布是只有软件包，还是只有配置文件或者全部都需要发布。
第22行，name，发布单名称，这个名称需要具有唯一性，我们的定义规则是，将当前时间精确到秒，再加上随机数来保证唯一性。这样用户一眼便知道是哪个日期发布的发布单。
第24~27行，app_name，这个发布单所对应的 App 的名称，它是以外键形式关联的；有了这个信息，通过 Django ORM，可以很方便地通过发布单，查找对应 App 数据表里的配置。
第28行，deploy_no，这个字段，表示发布单自创建以来，一共被发布了多少次。它主要用来追踪用户发布的实时日志读取及历史日志读取，在后面对于 Django Channels 实现 WebSocket 的实时日志读取，主要就是用的这个字段进入联动。
第29行，branch_build，本次发布单所对应的 Git 版本，每次编译软件时，这个信息会和 App里的 Git 信息组合，形成唯一定位的 Git 版本，Jenkins 会根据这些信息进行编译。
第30行，jenkins_number，这里主要记录的是 Jenkins 的编译次数，如果在编译过程中有任何问题，都可以快速定位到相关网页进行排错。
第31行，code_number，在 Jenkins 从 Git 拉取代码的过程中，会有一个唯一的 Git内部区别版本号，通过这个版本号，就可以定位 Git 上唯一的代码版本。
第32~35行，is_inc_tot，用于记录本次是全量还是增量发布，选择项已在上文说明。
第36~39行，deploy_type，用于记录本次发布的是配置，程序包还是项目的所有文件，选择项已在上文说明。
第41~44行，create_user，用于记录创建发布单的用户。
第45行，nginx_url，用于记录本次发布单生成的软件位于 Nginx 的哪个目录下。这个字段存在的意义是：由于我们不是用 Jenkins 同时做 CI/CD 工作，而只是作 CI 集成，那么 Jenkins 生成的软件包不会直接发布到服务器上进行启停操作，而是先把这个软件包放在一个 Nginx 服务器上。在发布单进行环境流转之后，才会在对应的环境上，用发布脚本中的 wget 将软件推送到对应的服务器上进行发布。这个作用，与 Docker 镜像的 Harbor 仓库类似，起一个软件仓库的作用，随时可以查找以前的安装包。
第46~50行，env_name，用于记录发布单当前处于哪个环节，注意，在刚刚创建发布单和进行软件编译时，还没有环境，所以这个字段要允许为空。
第51~55行，deploy_status，用于记录发布单当前处于哪一个声明周期，当它处于创建和编译时，是独立存在的周期，一旦它在环境流转之后，就可以和环境 env_name 组合，形成诸如 TEST 待发布，DEV 发布异常，PRD 发布中，PRD 发布完成等状态，给操作者一个清晰的信息。
