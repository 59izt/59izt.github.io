¶ RMAN 环境概述

恢复管理器 (RMAN) 是一个 Oracle 数据库客户端，可对您的数据库执行备份和恢复任务并自动管理您的备份策略。它极大地简化了备份、还原和恢复数据库文件。

RMAN 环境由用于备份数据的实用程序和数据库组成。 RMAN 的环境至少必须包括以下组件：

  • 一个目标数据库: RMAN 使用 TARGET 关键字连接到的 Oracle 数据库。目标数据库是 RMAN 在其上执行备份和恢复操作的数据库。RMAN 始终在数据库的控制文件中维护有关其对数据库的操作的元数据。RMAN 元数据被称为 RMAN 存储库。
  • RMAN 客户端: 一个 Oracle 数据库可执行文件，用于解释命令、引导服务器会话执行这些命令，并在目标数据库控制文件中记录其活动。 RMAN 可执行文件随数据库自动安装，通常与其他数据库可执行文件位于同一目录中。例如，Linux 上的 RMAN 客户端位于 $ORACLE_HOME/bin 中。

某些环境使用以下可选组件：

  • 快速恢复区: 数据库可以在其中存储和管理与备份和恢复相关的文件的磁盘位置。您可以使用 DB_RECOVERY_FILE_DEST 和 DB_RECOVERY_FILE_DEST_SIZE 初始化参数设置快速恢复区的位置和大小。
  • 介质管理器: RMAN 与顺序介质设备（如磁带库）交互所需的应用程序。介质管理器在备份和恢复期间控制这些设备，管理介质的加载、标记和卸载。介质管理设备有时称为 SBT（系统备份到磁带）设备。
  • 恢复目录: 一种单独的数据库模式，用于记录针对一个或多个目标数据库的 RMAN 活动。如果控制文件丢失，恢复目录会保留 RMAN 存储库元数据，从而在控制文件丢失后更容易恢复和恢复。数据库可能会覆盖控制文件中的旧记录，但 RMAN 会在目录中永远保留记录，除非用户删除这些记录。

本章介绍了如何在没有恢复目录或介质管理器的最基本配置中使用 RMAN。

¶ 显示默认的 RMAN 配置

RMAN 备份和恢复环境是为每个目标数据库预先配置的。即使您退出并重新启动 RMAN，该配置也是持久的并适用于此目标数据库上的所有后续操作。
RMAN 配置的设置可以指定备份设备、配置到备份设备的连接（称为通道）、影响备份策略的策略等。默认配置足以满足大多数用途。

¶ 备份数据库

使用 BACKUP 命令备份文件，RMAN 将数据备份到为请求的备份类型配置的默认设备。默认情况下，RMAN 在磁盘上创建备份。如果启用了快速恢复区，并且未指定 FORMAT 参数，则 RMAN 在恢复区中创建备份并自动为其指定唯一名称。
默认情况下，RMAN 创建备份集而不是映像副本。一个备份集由一个或多个备份片段组成，这些备份片段是以只有 RMAN 可以访问的格式编写的物理文件。多路复用备份集包含来自多个输入文件的块。RMAN 可以将备份集写入磁盘或磁带。

如果指定 BACKUP AS COPY，则 RMAN 会将每个文件复制为映像副本，这是在磁盘上创建的数据库文件的逐位副本。映像副本与使用操作系统命令（如 Linux 上的 cp 或 Windows 上的 COPY）创建的副本相同，但记录在 RMAN 存储库中，因此可供 RMAN 使用。您可以在数据库打开时使用 RMAN 制作映像副本。

¶ 常用的备份选项

BACKUP 命令包括许多控制备份输出的选项、参数和子句。下表列出了一些常用的备份选项。

选项 描述 示例

¶ 增量备份

如果指定 BACKUP INCREMENTAL，则 RMAN 会创建数据库的增量备份。增量备份捕获在上一次增量备份之后对数据库所做的块级更改。增量备份通常比完整数据库备份更小、更快。使用增量备份进行恢复比单独使用重做日志要快。

增量备份策略的起点是 0 级增量备份，它备份数据库中的所有块。级别 0 的增量备份与完整备份的内容相同，但是，与完整备份不同的是，级别 0 备份被视为增量备份策略的一部分。
1 级增量备份仅包含在上一次增量备份后更改的块。如果在运行 1 级备份时当前或父数据库化身中不存在 0 级备份，则 RMAN 会自动进行 0 级备份。

当 NOARCHIVELOG 数据库打开时，您不能进行增量备份，尽管您可以在持续关闭后装入数据库时​​进行增量备份

级别 1 的备份可以是累积增量备份，其中包括自最近的级别 0 备份以来更改的所有块，或差异增量备份，其中仅包括自最近的增量备份以来更改的块。默认情况下，增量备份是差异备份。

在恢复增量备份时，RMAN 使用 0 级备份作为起点，然后在可能的情况下根据 1 级备份更新更改的块，以避免一次重做一次重新应用更改。使用增量备份进行恢复不需要您付出额外的努力。如果增量备份可用，则 RMAN 在恢复期间使用它们。

¶ 进行增量更新备份

RMAN 增量更新备份功能是一种高效的增量备份策略。该策略具有以下主要特点：

  • 该策略需要一个 0 级数据文件副本作为基础。此副本具有系统定义或用户定义的标记。
  • 定期使用与 0 级数据文件副本相同的标记创建 1 级差异备份。 BACKUP FOR RECOVER OF COPY 命令指定增量备份应仅包含自最近一次增量备份以来更改的具有相同标记的块。
  • 增量备份会定期应用于 0 级数据文件副本。由于数据文件副本已更新为最近的更改，因此现在需要更少的介质恢复。

下表说明了与 FOR RECOVER OF COPY 一起使用的哪些选项来实施增量更新的备份策略。

备份选项 说明 示例
FOR RECOVER OF COPY WITH TAG ‘tag_name’ 使用 TAG 标识作为备份策略依据的数据文件副本的标签。RMAN 会自动为该副本的每个 1 级备份分配相同的标记，如果当前或父数据库化身中不存在具有指定标记的 0 级数据文件副本，则 RMAN 会创建具有指定标记的 0 级数据文件副本。 BACKUP INCREMENTAL LEVEL 1 FOR RECOVER OF COPY WITH TAG ‘incr_update’ DATABASE;
FOR RECOVER OF COPY DATAFILECOPY FORMAT ‘format’ 如果副本不存在，则指定 RMAN 在何处创建数据文件副本。如果向数据库添加新数据文件，则无需更改脚本，因为 RMAN 会自动创建增量备份例程所需的 0 级副本。 BACKUP INCREMENTAL LEVEL 1 FOR RECOVER OF COPY DATAFILECOPY FORMAT ‘/disk2/df1.cpy’ DATABASE;

¶ 使用 Data Recovery Advisor 诊断和修复故障

诊断和修复数据库问题的最简单方法是使用 Data Recovery Advisor。该 Oracle 数据库工具提供了用于诊断持续数据故障、向用户提供修复选项以及自动执行修复的基础架构。

数据恢复顾问概述

¶ 列出故障并确定修复选项

故障是 Health Monitor 检测到的持续数据损坏。示例包括物理和逻辑数据块损坏和数据文件丢失。每个故障都有一个故障优先级和故障状态。优先级可以是 CRITICAL、HIGH 或 LOW。状态可以是打开或关闭。

¶ 使用闪回数据库倒带数据库

您可以使用 Oracle 闪回数据库将整个数据库倒回过去的时间。与介质恢复不同，您无需恢复数据文件即可将数据库恢复到过去的状态。
要使用 RMAN FLASHBACK DATABASE 命令，您的数据库必须先前已配置为生成闪回日志。此配置任务在闪回数据库中进行了描述。闪回数据库的工作原理是回滚对运行命令时存在的数据文件的更改。您不能使用该命令来修复媒体故障或丢失的数据文件。
发出 FLASHBACK DATABASE 时，必须装入数据库。如果您之前创建了一个还原点，那么如果它在闪回数据库窗口内，您可以闪回到这个还原点。
